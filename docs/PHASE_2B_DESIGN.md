\# Phase 2B 設計書：保護者LINE\_USER\_ID 収集システム



\## 1. 目的



全保護者のLINE\_USER\_IDを最速で収集し、引き落とし通知のLINE移行を完了させる。



\## 2. 動機



\- マイ活アプリ側でLINE Login OAuth問題が解決し、ID収集の仕組みは稼働中

\- しかし全保護者の登録は完了していない

\- LINE未登録の保護者には引き落とし通知が届かず、不利益となる

\- ID収集を加速し、全保護者への確実な通知体制を構築する



\## 3. 2方式併用



保護者の状況に応じて、2つの登録方式を提供する。どちらの方式で登録しても、最終的にFamiliesシートの「保護者LINE\_USER\_ID」列に書き込まれる。



\### 方式A：LINE Login OAuth（個別URL方式）



\*\*概要\*\*：保護者が個別URLをタップ → LINE認証 → 自動紐付け



\*\*フロー\*\*：

1\. ふくちさんが管理画面で「藤本様の登録URL発行」ボタンを押す

2\. システムが8文字英数字のトークン（例：xK7p9BdF）を生成

3\. Tokensシートに「トークン xK7p9BdF → 家族ID F001 → 状態：有効」を記録

4\. ふくちさんにURLが表示される（例：https://k-acdm.github.io/billing-line/register.html?t=xK7p9BdF）

5\. ふくちさんがURLをコピーし、保護者にLINE個別チャット等で送る

6\. 保護者がURLをタップ

7\. 登録画面が開き、「LINEでログイン」ボタンが表示される

8\. 保護者がボタンを押す → LINE認証 → 「藤本様として登録します」表示

9\. 「登録する」ボタンで確定

10\. システムがLINE\_USER\_IDをFamiliesシートに書き込み、トークンを使用済にする

11\. 保護者に「登録完了」画面表示



\*\*メリット\*\*：

\- 保護者が名前入力不要（誤入力なし）

\- 自動紐付けでふくちさんの手間が最小

\- マイ活で既に動いている仕組みを流用できる



\*\*対象保護者\*\*：スマホでURLを開ける環境にある人



\### 方式B：公式LINEメッセージ受信



\*\*概要\*\*：保護者が公式LINEに何か送信 → 管理画面でふくちさんが手動紐付け



\*\*フロー\*\*：

1\. ふくちさんが公式LINEから一斉配信で「『登録』とメッセージかスタンプを送ってください」と案内

2\. 保護者が公式LINEに何か送信（テキスト・スタンプ・画像なんでも可）

3\. Webhookがメッセージを自動受信し、IncomingMessagesシートに記録

&#x20;  - LINE\_USER\_ID

&#x20;  - 表示名

&#x20;  - プロフィール画像URL

&#x20;  - ステータスメッセージ

&#x20;  - 受信日時

&#x20;  - メッセージ種別・本文

4\. ふくちさんが管理画面の「未紐付けメッセージ一覧」を確認

5\. アイコン・表示名・本文・受信時刻から「誰のメッセージか」を判断

6\. 本文に苗字が含まれる場合、システムが自動推定して候補を出す

7\. ふくちさんがプルダウンで家族を選択 → 「紐付け確定」ボタン

8\. システムがLINE\_USER\_IDをFamiliesシートに書き込み、IncomingMessagesシートの該当行を「紐付け済」にする



\*\*メリット\*\*：

\- 保護者の手間が極小（スタンプ1個でOK）

\- スマホ操作に不慣れな保護者でも実行可能

\- URLを開けない環境でも登録できる



\*\*対象保護者\*\*：方式Aが難しい人、スタンプ送信で済ませたい人



\## 4. データ構造の追加



既存6シートに加え、以下2シートを追加する。



\### 4-1. Tokensシート（方式A用）



| 列名 | 型 | 説明 |

|------|-----|------|

| トークン | string | 8文字英数字（例：xK7p9BdF） |

| 家族ID | string | このトークンに紐付く家族 |

| 発行日時 | datetime | |

| 状態 | string | 有効 / 使用済 / 無効化 |

| 使用日時 | datetime | 保護者が登録完了した時刻 |



\### 4-2. IncomingMessagesシート（方式B用）



| 列名 | 型 | 説明 |

|------|-----|------|

| 受信日時 | datetime | |

| LINE\_USER\_ID | string | Webhookで取得 |

| 表示名 | string | LINEプロフィール名 |

| プロフィール画像URL | string | アイコン画像のURL |

| ステータスメッセージ | string | あれば |

| メッセージ種別 | string | text / sticker / image / video / audio / location / 他 |

| メッセージ本文 | text | textのみ。他は種別名を記録 |

| 紐付けステータス | string | 未紐付け / 紐付け済 / 無視 |

| 紐付け家族ID | string | 紐付け後に記録 |

| 紐付け日時 | datetime | 紐付け後に記録 |

\## 6. 自動推定ロジック（方式B）



未紐付けメッセージの本文に、Familiesシートの「宛名」から「様」を除いた苗字（例：「藤本」）が含まれる場合、その家族を候補1位として自動表示する。



複数の苗字がマッチする場合、最初にマッチしたものを優先する（後で改善可能）。



\## 7. 実装順序



ID収集を最速で動かすため、以下の順番で実装する：

Step 1：管理画面の骨格＋未登録家族リスト

\- GitHub Pages の admin/index.html

\- GAS側のWebアプリエンドポイント（家族一覧取得）

\- 一覧表示・CSVエクスポート

Step 2：個別URL発行機能（方式A用）

\- Tokensシート作成

\- GAS側でトークン生成・記録機能

\- 管理画面に「URL発行」ボタン

\- URL表示モーダル

Step 3：保護者登録画面（方式A用）

\- GitHub Pages の register/index.html

\- トークンからの家族特定処理

Step 4：LINE Login OAuth処理（方式A用）

\- マイ活のコードを流用

\- コールバック処理

\- Familiesシートへの書き込み

Step 5：Webhook受信（方式B用）

\- IncomingMessagesシート作成

\- GAS側のdoPost()エンドポイント

\- LINE Webhook URL設定

Step 6：未紐付けメッセージ画面＋紐付け確定（方式B用）

\- 管理画面に未紐付け一覧

\- 自動推定ロジック

\- 紐付け確定処理



\*\*判断指針\*\*：Step 1〜4 で方式Aが動けば ID収集を開始できる。Step 5〜6 は方式Aで取れない保護者向け。



\## 8. 関連する技術的判断



\### 8-1. トークン仕様

\- 8文字英数字（大小文字+数字）= 約2.8兆通り

\- Privateリポジトリのためソース漏洩リスクなし

\- 一度使われたトークンは「使用済」状態にし、再利用不可



\### 8-2. 認証

\- 管理画面はふくちさん専用 → マイ活と同じ講師ログイン方式を流用

\- 保護者登録画面は認証不要（トークン自体が認可となる）



\### 8-3. LINE Channel

\- 既存の春日部アカデミー公式LINE（@pwg1825g）を使用

\- Messaging API のChannel Access Token をスクリプトプロパティに登録

\- Webhook URL を GAS の Webアプリ URL に設定
